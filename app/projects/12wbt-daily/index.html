Background

12WBT Daily is a native iOS app designed to help members follow their fitness and nutrition programs on-the-go. It allows them to track their daily calorie consumption and expenditure, also allowing them to access their daily workout and meal plans via a touch-friendly interface. I was part of an agile team involved in the conception and implementation of the app.

Initially the client had a PhoneGap application which was slow and only functioned as a wrapper for the website. Understandably, they wanted it to run fast and smooth - so we decided to rebuild it natively using Swift.

Network latency

As the application is API-driven, we needed to find a way to make sure network latency does not affect the perception of speed when fetching and writing data, as users could be at a gym with poor mobile reception or completely offline.

We managed to build a caching system which works well with our API. The application handles various failure/conflict scenarios so that your data is automatically synced with API when possible. Stale caches were flushed by checking the relevant HTTP headers. Assets were also cached on disk and in memory so that they could be retrieved quickly.

UI performance

The core UI allows the user to swipe horizontally between different days. Each day or screen could also be swiped vertically to reveal different nutritional and fitness recommendations. We used a combination of UIPageViewController and UICollectionView to ensure views are recycled and memory was allocated correctly.

The day view acts as the central navigation hub, allowing users to drill deep and interact with their content. It required some ViewControllers to be pushed into the navigation stack, others to be presented modally.

Code structure

We decided to use the composite and delegate pattern to structure our code and separate concerns, so that various design requirements could be fulfilled and changes in the future do not require a major refactor.

Business and data manipulation logic was largely handled in model classes. This allowed us to easily query data and display it in different format - for example, the summary view displayed a memberâ€™s fitness data as an average, but in the detail view, the same data needed to be displayed as a Before and After table.

BDD testing

Coming from JavaScript and Ruby development background, we always like to use BDD testing frameworks to describe our tests, such as Jasmine and RSpec. Therefore, we decided to use Quick/Nimble instead of XCTest. It helped us to focus on identifying expected behaviours and contexts instead of testing implementation details.
